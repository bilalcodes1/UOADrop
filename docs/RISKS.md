# المخاطر والوقاية والاستجابة

> **ملاحظة قراءة:** هذا المستند يشمل مخاطر **التطبيق الحالي المحلي** وكذلك مخاطر **المراحل القادمة للـ online/cloud**. أي قسم يتحدث عن Supabase أو Vercel أو Email/Telegram يُقرأ على أنه **مستقبلي** ما لم يُذكر خلاف ذلك.

هذا المستند يوثّق كل مصدر فشل محتمل في UOADrop، مع الوقاية والاستجابة المخطط لها. يُراجع كل 6 أشهر أو بعد أي حادثة.

---

## منهجية التصنيف

كل مخاطرة مصنّفة بثلاثة أبعاد:
- **الاحتمال**: مرتفع / متوسط / منخفض
- **الأثر**: حرج (يوقف الخدمة) / متوسط (يبطّئها) / خفيف (إزعاج)
- **الاستجابة**: preventive (قبل الحدوث) / reactive (بعد الحدوث)

## 1. 🏢 مخاطر مزوّد الخدمة

### 1.1 Supabase Free Tier Pause
- **الاحتمال**: مرتفع (بعد 7 أيام خمول)
- **الأثر**: حرج للـ online — يوقف صفحة بلال
- **الوقاية**: Keep-Alive 3 طبقات (desktop + cron-job.org + GitHub Actions) → ping يومي
- **الاستجابة**: لو pause صار، فك من Supabase dashboard (خلال 90 يوم) → الطلبات محفوظة

### 1.2 Supabase bandwidth limit (2GB/شهر على free)
- **الاحتمال**: منخفض (يحتاج ~1000 طلب/شهر)
- **الأثر**: حرج مؤقتاً
- **الوقاية**: مراقبة usage من Dashboard + إشعار عند 80%
- **الاستجابة**: ترقية لـ Pro $25/شهر، أو هجرة لـ self-hosted

### 1.3 Vercel يغيّر pricing أو يوقف free tier
- **الاحتمال**: منخفض
- **الأثر**: حرج للـ online
- **الوقاية**: المشروع portable — يشتغل على Netlify أو Cloudflare Pages بدون تعديل كود
- **الاستجابة**: نقل deployment خلال ساعة

### 1.4 Domain `uoadrop.app` ينتهي
- **الاحتمال**: متوسط (كل سنة)
- **الأثر**: حرج للـ online
- **الوقاية**: Auto-renew + تذكير قبل 60 يوم + 30 يوم
- **الاستجابة**: تجديد فوري، الـ QR الأخضر يرجع يشتغل

### 1.5 Supabase محجوب في العراق مستقبلاً
- **الاحتمال**: منخفض
- **الأثر**: حرج للـ online
- **الوقاية**: Self-host على VPS أوروبي ($5/شهر) كخيار احتياطي
- **الاستجابة**: هجرة DNS → VPS خلال يوم

## 2. ⚡ مخاطر البنية التحتية

### 2.1 انقطاع الكهرباء
- **الاحتمال**: مرتفع (العراق)
- **الأثر**: متوسط — الخدمة تتوقف حتى رجوع الكهرباء
- **الوقاية**: UPS صغير ($50) للابتوب + الراوتر = 30 دقيقة تشغيل
- **الاستجابة**: النظام يرجع تلقائياً عند الكهرباء (auto-start + MAC binding)

### 2.2 انقطاع الإنترنت
- **الاحتمال**: مرتفع
- **الأثر**: خفيف — الـ Offline يعمل بالكامل، فقط بلال يتأثر
- **الوقاية**: النظام Offline-first بالتصميم
- **الاستجابة**: سعد يكمل شغله مع ملاك بدون مشاكل

### 2.3 راوتر تعطّل
- **الاحتمال**: منخفض (TL-WR940N معمّر)
- **الأثر**: حرج للـ offline
- **الوقاية**: راوتر احتياطي مشابه + backup إعدادات (`config.bin`)
- **الاستجابة**: استبدال + استعادة config = 5 دقائق

### 2.3.5 سعد يغلق غطاء اللابتوب ويخرج (C8)
- **الاحتمال**: متوسط (نهاية الدوام / استراحة غداء)
- **الأثر**: حرج للطلاب offline — النظام يدخل sleep وينقطع Fastify + Wi-Fi hotspot
- **الوقاية**: ضبط `lidaction = 0` على Windows + `pmset disablesleep 1` على Mac (راجع `SETUP.md` §2.4.1) + banner تحذيري في التطبيق لو الإعداد غير صحيح
- **الاستجابة**: طالب يتصل فيزيائياً بالمكتبة لو لم يستطع الرفع

### 2.4 لابتوب سعد تعطّل
- **الاحتمال**: منخفض
- **الأثر**: حرج للـ offline والـ dashboard
- **الوقاية**: Daily backup للـ SQLite + الملفات على external drive أو iCloud
- **الاستجابة**: لابتوب بديل + استعادة backup + تحديث MAC binding في الراوتر = 15 دقيقة

### 2.5 طابعة تعطّل
- **الاحتمال**: متوسط
- **الأثر**: متوسط — الطلبات تتراكم لكن تبقى محفوظة
- **الوقاية**: طابعة احتياطية
- **الاستجابة**: تبديل + إعادة ضبط الطابعة الافتراضية في التطبيق

## 3. 💾 مخاطر التخزين

### 3.1 قرص لابتوب ممتلئ
- **الاحتمال**: متوسط بعد شهور من الاستخدام
- **الأثر**: حرج — ما يقدر يستلم طلبات جديدة
- **الوقاية**: Auto-delete للملفات المكتملة بعد 7 أيام + تنبيه عند 80% ممتلئ
- **الاستجابة**: تنظيف يدوي من مجلد التطبيق

### 3.2 SQLite database فسدت
- **الاحتمال**: منخفض جداً (SQLite معمّر جداً)
- **الأثر**: حرج — فقدان الطلبات النشطة
- **الوقاية**: WAL mode + integrity check عند كل تشغيل + daily backup
- **الاستجابة**: استعادة من backup (أقصى خسارة = طلبات آخر 24 ساعة)

### 3.3 ملف ضخم جداً يعطّل الرفع
- **الاحتمال**: متوسط
- **الأثر**: متوسط
- **الوقاية**: حد 50MB/ملف، 200MB/طلب، مع رسالة واضحة
- **الاستجابة**: الطالب يضغط الملف أو يقسّمه

## 4. 🔐 مخاطر أمنية

### 4.1 طالب يرفع ملف خبيث (.exe, .bat, macro, rename trick) — C7
- **الاحتمال**: منخفض (طلاب عاديون)
- **الأثر**: متوسط إلى حرج لو سعد فتحه
- **الوقاية بطبقتين** (C7):
  1. **Whitelist الامتداد**: `pdf, docx, doc, pptx, ppt, xlsx, xls, txt, jpg, png` فقط.
  2. **Magic-byte validation** عبر `file-type` npm — يقرأ أول 4KB من الملف ويتحقق من النوع الحقيقي. يرفض `evil.exe` المُعاد تسميته إلى `report.pdf`.
- **الاستجابة**: سعد يفتح الملف بالتطبيق الافتراضي (Preview/Word) — التطبيقات الموثوقة تحمي من macros الخبيثة افتراضياً

### 4.2 DDoS / spam على الصفحة
- **الاحتمال**: منخفض
- **الأثر**: متوسط
- **الوقاية**: Rate limit 5 طلبات/دقيقة لكل IP + CAPTCHA بعد 3 متتالية
- **الاستجابة**: منع IP من التطبيق يدوياً

### 4.3 سرقة Supabase API keys
- **الاحتمال**: منخفض (anon key معروض للجمهور أصلاً)
- **الأثر**: متوسط — RLS يحمينا
- **الوقاية**: Row Level Security صارمة — anon فقط يقدر INSERT
- **الاستجابة**: rotate keys من Supabase dashboard + redeploy

### 4.4 سعد يفقد password الـ dashboard
- **الاحتمال**: منخفض
- **الأثر**: خفيف
- **الوقاية**: password manager (1Password/Keychain)
- **الاستجابة**: reset من Supabase dashboard

### 4.5 تسرّب بيانات طلاب (أسماء + ملفات)
- **الاحتمال**: منخفض
- **الأثر**: متوسط (خصوصية)
- **الوقاية**: auto-delete بعد 7 أيام + عدم تخزين معلومات حساسة (أرقام هوية، إلخ)
- **الاستجابة**: إبلاغ الطلاب + review الأمان

## 9. 📨 مخاطر الإشعارات (بلال Online فقط)

### 9.1 تجاوز حد Resend (3,000 email/شهر)
- **الاحتمال**: منخفض (تقديرنا 900 email/شهر)
- **الأثر**: متوسط — إشعارات Email تتوقف، Telegram يستمر
- **الوقاية**: Rate limit 5 طلبات/ساعة/email + مراقبة usage في Health Dashboard + تنبيه عند 80%
- **الاستجابة**: ترقية Resend ($20/شهر = 50k email) أو هجرة لـ AWS SES

### 9.2 فشل Telegram Webhook
- **الاحتمال**: منخفض
- **الأثر**: حرج لإشعارات Telegram — ما يستلم `/start` من الطلاب الجدد
- **الوقاية**: `setWebhook` مع `max_connections` وإعادة تسجيل تلقائية كل 24 ساعة + `getWebhookInfo` للفحص
- **الاستجابة**: إعادة تسجيل webhook يدوياً + فحص Edge Function logs

### 9.3 تسرّب RESEND_API_KEY أو TELEGRAM_BOT_TOKEN
- **الاحتمال**: منخفض (محفوظة في Supabase env فقط)
- **الأثر**: حرج — مهاجم يستطيع إرسال spam باسم المشروع
- **الوقاية**: لا ترفعها لـ git أبداً + `.env` في `.gitignore` + Supabase Secrets vault
- **الاستجابة**: rotate keys فوراً من Resend + BotFather (`/revoke`) + redeploy

### 9.4 Email bounces تضر domain reputation
- **الاحتمال**: متوسط (طلاب يُدخلون email غلط)
- **الأثر**: متوسط — Gmail/Outlook يحوّلون رسائلنا للـ spam
- **الوقاية**: Zod validation صارم + SPF/DKIM/DMARC records على `uoadrop.app` + Resend يتعامل مع bounces تلقائياً
- **الاستجابة**: مراقبة bounce rate في Resend dashboard، حذف emails الفاشلة بعد bounce واحد

### 9.5 الطالب يحجب البوت (`/block`)
- **الاحتمال**: منخفض
- **الأثر**: خفيف — إشعارات Telegram تفشل لهذا الطالب فقط
- **الوقاية**: لا شي — حق الطالب
- **الاستجابة**: عند error `403 Forbidden: bot was blocked` → حذف `telegram_chat_id` تلقائياً + سقوط تلقائي لـ Email

### 9.6 Telegram Bot API محدود في العراق
- **الاحتمال**: منخفض
- **الأثر**: حرج إذا حصل
- **الوقاية**: مراقبة + الحفاظ على قناة Email كـ redundant
- **الاستجابة**: استخدام proxy في Edge Function أو الاكتفاء بـ Email

### 9.7 Edge Function timeout أثناء الإرسال
- **الاحتمال**: منخفض (Supabase Edge = 150s limit)
- **الأثر**: خفيف — إشعار واحد يتأخر
- **الوقاية**: إرسال parallel عبر `Promise.allSettled` + timeout 10s لكل API call
- **الاستجابة**: Retry queue يلتقط الفاشلات تلقائياً

### 9.8 DB trigger فشل أو مفصول
- **الاحتمال**: منخفض
- **الأثر**: حرج — لا إشعارات أبداً
- **الوقاية**: Health check يومي يتحقق أن آخر `notifications_log` < 24 ساعة
- **الاستجابة**: إعادة إنشاء trigger + retry آخر 24 ساعة يدوياً

### 9.9 تسرّب بيانات الاتصال (email + chat_id)
- **الاحتمال**: منخفض
- **الأثر**: متوسط (خصوصية)
- **الوقاية**: RLS صارم — فقط سعد والبوت يقرأون + auto-delete بعد 30 يوم
- **الاستجابة**: إبلاغ الطلاب + مراجعة RLS policies

### 9.10 طالب spam يرفع طلبات وهمية لإرسال emails
- **الاحتمال**: منخفض
- **الأثر**: متوسط (استهلاك quota + ضرر reputation)
- **الوقاية**: Rate limit 5 طلبات/ساعة/IP + CAPTCHA بعد 3 طلبات + Resend يرصد الأنماط
- **الاستجابة**: حظر IP + الإبلاغ لـ Resend

## Runbook — إجراءات الطوارئ

### حادثة: "النظام لا يستلم طلبات جديدة"
1. تحقق من الـ Health Dashboard — ما المؤشر الأحمر؟
2. فحص IP → إذا ≠ 192.168.0.100 → أعد الاتصال بالـ Wi-Fi.
3. فحص Firewall → تأكد أن UOADrop مسموح.
4. فحص Port → أعد تشغيل التطبيق.
5. لو كل شي سليم → اختبر من موبايل آخر.

### حادثة: "Supabase paused"
1. افتح Supabase Dashboard.
2. اضغط **Unpause**.
3. تحقق من Keep-Alive logs في التطبيق.
4. أضف cron-job.org إذا لم يكن مضافاً.

### حادثة: "اللابتوب تعطل"
1. أحضر لابتوب بديل.
2. نصّب UOADrop.
3. استعد SQLite backup من external drive.
